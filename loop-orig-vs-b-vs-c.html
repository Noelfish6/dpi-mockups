Case 1 · C4 · 3 steps

No loop · No filter_by
AskBo (Redirected): Agent Request
Among users who opened an AskBo chat and explicitly requested a human agent between June 10 and June 16, 2026, approximately 1 in 3 (31.7%, 3,444 of 10,872 devices) were redirected to self-service rather than connected to a live agent. What made this pattern structurally distinctive was that the routing decision did not appear to be driven by the agent connection request itself — 91.5% of redirected users sent an agent connection message with no topic tag, a rate nearly identical to the 82.3% seen in the escalated comparison group. Instead, the prior conversation topic before the agent request was the strongest observable differentiator: sessions preceded by an Information Request / Information Verification exchange showed a 39.7% redirect rate, while sessions preceded by a Booking management / Refund exchange showed a 4.4% redirect rate. After the agent request was sent, the bot generated an average of 2.44 additional self-service prompts before executing the redirect, with 89.3% of redirected sessions receiving a 'Deflect & Offer Help' response at the redirect event — a consistent pattern stable across both analysis windows.

Case 2 · E2 · 7 steps · Bot action × 2

Single loop · No filter_by
AskBo (Agent Connection): Refund Retried, Process Started
Users who retried a Refund request and progressed through the bot's process initiation step before reaching an Information Verification exchange represent a segment that had already committed to a multi-step resolution attempt. Despite completing this extended bot engagement, the majority requested a human agent within seconds of the Information Verification message — suggesting the bot flow at this stage does not provide sufficient resolution guidance for this cohort. The pattern separates into three structurally distinct trajectories based on post-verification engagement depth: a rapid exit to agent request, an extended information exchange that reaches full resolution, and a prolonged high-frustration loop that ends in system escalation.

Case 3 · E3 · 9 steps · Refund × 3 + Pending × 2

Multi-loop · No filter_by
AskBo (Agent Connection): Refund Restarted, Process Started
Users who initiated a refund request, received an unexpected conversation reset mid-session, and persisted through a full second refund attempt before reaching process initiation represent a pattern of compounded friction within a single bot interaction. Unlike users who encountered refund delays without a session restart, this cohort faced a doubled accumulation of waiting steps across the restarted conversation, correlating with a meaningfully higher rate of human agent escalation than comparable refund journeys without a restart event. The defining structural feature of this pattern is the bot's sequencing at the process initiation stage — where confirmation of refund start is followed near-instantaneously by a waiting-state message, leaving users with unresolved expectations that the subsequent verification step does not appear to address.

C — Time encoding variants (advanced, on Case 3)

Synthetic Δt per edge · purely illustrative
Why encode time?
Same 9-step Case 3 pattern, now with per-edge wait times. The structural layout (loops, fold-backs) is unchanged — we only add a second channel that shows how long the user spent between steps. Two variants below trade off differently.
Synthetic times: 1→2 5s · 2→3 3s · 3→4 2s · 4→5 5s · 5→6 15s · 6→7 5s · 7→8 25s · 8→9 60s · total ≈ 120s
C · distancehorizontal position encodes cumulative time · swimlane dropped

Quantitative + ordinal — distance maps directly to seconds; readers can measure, not just rank. Long waits become visually long gaps.

Friction shape is obvious — three big gaps after step 5 jump out as the dominant pattern; the early refund loop compresses into a tight cluster.

Swimlane lost — repeated steps no longer share a column; loops must be re-read off the repeat arcs and color coding.

Width depends on data — a single 10-minute gap can stretch the whole viz horizontally; needs log-scale or clipping when Δt is heavy-tailed.

C · thicknessstroke-width encodes Δt · swimlane preserved

Swimlane preserved — loop structure and step columns are unchanged; time is layered onto the existing C without rearranging anything.

"Stuck-then-escalates" reads at a glance — 25s into Pending, then 60s before Info verify shows up as thickening edges.

Quantitative comparison is hard — stroke width is a weak channel for precise duration; readers can rank but not measure. Numeric labels mitigate but clutter.

Small Δt vanishes — 2s vs 5s edges look nearly identical; thickness saturates at both ends of the scale.

C · swim + timey = step lane (labeled) · x = cumulative time (9 px / sec) · horizontal = Δt on a lane (color = lane) · 90° corner = lane change · filter_by pill lives in the widened left gutter with a bracket into the constrained lanes

Same step = same lane, perfectly aligned — Refund req (2, 4, 6) share one horizontal lane; Pending (5, 8) share another. The swimlane property is fully preserved, just rotated 90° from the original column layout.

x = cumulative time, no compromises — every node's x position is its real timestamp; horizontal distance between any two nodes is their actual Δt.

Horizontal = time on a lane, 90° corner = lane change — each solid horizontal's length equals Δt and its color matches the lane it sits on (cyan on Refund req, purple on Pending, grey elsewhere); the dashed vertical is just a connector with no time meaning. Loop-backs read as a dashed jump upward into a colored lane, no extra coloring needed.

Vertical real estate scales with unique step count — 6 steps → 6 lanes here. Patterns with 10+ unique steps need a taller chart or a scrollable y-axis.

Heavy-tail Δt still blows up width — the 60s gap dominates the right half of the viewBox; same log-scale or clipping consideration as Distance.

Which to pick? Thickness when loop structure is the headline and Δt is supporting context — the swimlane carries the narrative, edge weight adds texture. Swim+time is the strongest hybrid — same-step lanes (swimlane) + real time axis (distance); loops read as visits to the same row, time reads off the x-axis. Distance when you want the most compact single-row time axis and don't mind reading loops off arcs + color. They are not mutually exclusive: a UI could let the user toggle between these views on the same pattern.
The real problem with the original: The current PatternStepViz actually supports repetition role and milestone (pill + amber + green three-color rules) — but the Pattern detail page's adapter chain segment_steps[] → stepsToFlowData → flowDataToSequence marks every node as phase='shared' and skips repeatCount, so repetition structure and outcome are never passed to the viz.
→ No need to reinvent the viz; both B and C can be implemented as a "loop-aware adapter" on top of the same PatternStepViz component.
B summary: Linear timeline, visible loop arcs, intuitive "stuck" semantics for dialog-style patterns. With complex cases, arcs need to be layered and color-coded to stay readable. filter_by is shown with a node badge plus a bottom time-difference bracket overlay, without breaking the main flow.
C summary: Linear stretches stay on one row; a new row is only added when a step folds back to a column on its left. Returning to the same column = a loop, unambiguously; extending to N loop groups needs no new rules. Height grows with the number of fold-backs, not with step count — a 9-step pattern with 3 folds is 4 rows, a linear pattern is 1 row. filter_by uses a top pill plus a hanging bracket anchored to the adjacent step columns.
